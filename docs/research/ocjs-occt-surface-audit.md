## Executive Summary

**Matrix coverage verdict: substantively complete, with one uncovered shape and two speculative rows.**

The 37-row matrix covers every C++ source shape observed in the production OCJS binding surface (5,324 generated `.cpp` files, 2,117 distinct embind `class_<>` registrations) **except** `std::initializer_list<T>` constructor parameters used by NCollection list/sequence auto-generated bulk-init constructors (61 instances). That shape is added as row 38.

Two existing rows have **zero production instances** in current OCCT V8 source and should be flagged as speculative:

- **Row 23** (defaulted handle with non-null sentinel default `= Handle_Foo_Default()`) — exhaustive header scan of `repos/opencascade.js/deps/OCCT/src/**/*.hxx` returned 0 matches for `= Handle_X_*Default*()`, `= new Handle_X_…()`, or `= NS::SomeStaticHandle()` shaped non-null handle defaults. Every handle default in OCCT today is `= Handle()` (null), which is row 3.
- **Row 37** (reference-default to long-lived singleton `T& foo = singleton()`) — guarded by the R6-A static_assert at bindgen time; no production instances escape the gate.

The two rows that drive architectural decisions both have non-zero confirmed production instances:

- **Row 8 (sub-2b, degenerate sibling constructors)**: **19 instances across 14 distinct classes in 7 modules** (BRepGProp_Face is the canonical smoking gun; 13 additional classes; module breakdown below).
- **Rows 21 + 22 (genuine source-level `std::optional<T>` return + parameter)**: **38 OCCT public headers** use source-level `std::optional<T>` in fields, returns, or parameters. Production-emitted instances confirmed for both (e.g. `BOPDS_Interf::GetIndexNew() → std::optional<int>` for row 21; `BRepGraph_ParentExplorer` and `BRepGraph_ChildExplorer` constructors take `const std::optional<BRepGraph_NodeId::Kind>& theAvoidKind` for row 22). The canonical `std::optional<T>` domain — rows {3, 4, 5, 21, 22} — remains intact.

**Sub-2b detector scope (OQ2):** the production surface does NOT exhibit a single instance of a sub-2b pattern that spans inheritance, template-collapsed instantiations, or ADL/free-function namespaces. All 19 sub-2b instances are co-defined inside a single C++ source class. Rule 2's existing same-class detector scope is sufficient. The audit recommends recording this as an explicit invariant in the policy with a re-audit checkpoint if OCCT adds future cross-class same-name overloads.

### What was scanned

### Sampling vs exhaustive

- **Exhaustive** for sub-2b enumeration (row 8), genuine source-level `std::optional<T>` (rows 21, 22), non-null handle defaults (row 23), inheritance fan-out (`, base<>` counts), and operator/free-function existence (rows 29, plus the global "OCJS surfaces 0 namespace-scoped free functions and 0 operator overloads" finding).
- **Approximate** for very-high-count rows (row 6 multi-overload arity-only at ~760 method-groups, row 9 same-arity class-typed at ~1,226, row 28 NCollection template instantiations at ~890 binding files). These rows do not drive architectural decisions; instance counts are reported as-of-extracted from the aggregated bindings file.

### Confidence calibration

- **High confidence**: sub-2b enumeration (deterministic detector), non-null handle default zero-count (OCCT header scan), initializer_list shape (61 confirmed direct matches), 0 namespace-scoped free functions, 0 operator overloads in surface, inheritance fan-out 3,854/2,117 class registrations.
- **Medium confidence**: per-T `register_optional<>` breakdown (60 distinct T types, 1,448 total registrations — but the same T may be registered in multiple files for default-arg expansion vs genuine optional vs both, so per-row attribution is by usage pattern not registration count).
- **Lower confidence**: row 7 sub-2a precise count (heuristic; the BRepMesh_IncrementalMesh canonical case is confirmed, but the heuristic also flags ~190 false positives from NCollection auto-generated constructor variants — a precise detector would require typed AST analysis).

### Sub-2b Detection Algorithm (Validated)

```text
For each class with ≥2 constructors:
  for each pair (A, B) where len(B.params) == len(A.params) + 1:
    if B.params[1:] == A.params  AND  A.params[-1] is std::optional<...>:
      flag as sub-2b: B subsumes A; A would shadow B via libembind optional-wildcard short-circuit
```

This algorithm runs at bindgen-emit time per the policy's rule 2. Implementation note: the equality `B.params[1:] == A.params` MUST normalize qualifier whitespace (`const T &` vs `const T&`) and `opencascade::handle` vs `occ::handle` aliasing. The current production bindings use `occ::handle` consistently inside `std::optional<...>` (per the typedef-cache normalization fix recorded in `learned-runtime.mdc`).

## Inheritance + Template-Collapsed Surface Findings (OQ2)

OQ2 asked whether the sibling-aliasing detector (rule 2) needs to scope across inheritance / template / ADL. The audit's findings:

### Inheritance

- **3,854 `, base<>` class registrations across 2,117 classes** (1.82 base-registrations per class average — many classes participate in inheritance hierarchies with multiple inherited methods).
- **Sample inheritance chain**: `TopoDS_Shape` → `TopoDS_Face` / `TopoDS_Edge` / `TopoDS_Solid` / `TopoDS_Vertex` / `TopoDS_Wire` / `TopoDS_Shell` / `TopoDS_CompSolid` / `TopoDS_Compound`. Inspection shows derived classes each emit ONLY `.constructor<>()` (default ctor) and inherit all methods from `TopoDS_Shape` via the embind `base<>` mechanism. **No derived class redefines a same-name method or constructor of the base.** Therefore inheritance-spanning sub-2b is structurally absent from this hierarchy.
- **Sample inheritance chain (NCollection)**: `NCollection_BaseSequence` → `NCollection_Sequence_*`; `NCollection_BaseMap` → `NCollection_DataMap_*`. Each derived class is a separate concrete instantiation; inherited methods are bound per-instantiation (no cross-template same-name overload aliasing in production).
- **Verdict**: rule 2's same-class detector scope is **sufficient** for the current OCCT surface. Cross-inheritance sub-2b would require a derived class to redefine `C(T1, T2 = D)` while the base defined `C(T2 = D)` — and we found zero instances of this in the production surface.

### Template-collapsed instantiations

- **890 NCollection binding files** under `NCollection_*.cpp`. Each NCollection template instantiation (e.g. `NCollection_Sequence_TopoDS_Shape`, `NCollection_DataMap_int_handle_Standard_Type`) emits its own `EMSCRIPTEN_BINDINGS` block with its own per-class detector scope.
- **No cross-instantiation same-name overload aliasing observed.** Each instantiation is bound under a distinct JS class name (the typedef chain becomes part of the bound name — e.g. `NCollection_Array1_AppParCurves_MultiBSpCurve`), so there is no "JS class" that aggregates two NCollection instantiations into a single dispatch table.
- **Verdict**: rule 2's per-class detector scope is **sufficient**. Template-collapsed surface is structurally disjoint at the JS dispatch level.

### ADL / free-function namespaces

- **0 namespace-scoped `function("…", &foo)` registrations in the entire surface.** Every OCCT free function (e.g. `BRepTools::Read`, `BRepLib::BuildCurve3d`) is wrapped inside a utility class as a `class_function`. Total `class_function` registrations: **7,034**.
- **Verdict**: ADL-spanning sub-2b is structurally **impossible** in the current emission. The "utility class" pattern bounds each free-function group to a single embind class.

### Summary

The audit confirms rule 2's same-class detector scope is correct for the current production surface. **No expansion required.** The policy SHOULD record this as a positive invariant with a follow-up audit checkpoint if any of the structural assumptions change (e.g. if a future OCJS feature switches from `class_function` to namespace-scoped `function()`, or if NCollection instantiations are merged under a shared JS class name).

## Uncovered Shapes

### New Row 38 — `std::initializer_list<T>` constructor parameters

**Source pattern**: NCollection list/sequence/array auto-generated bulk-init constructors with a `std::initializer_list<T>` first parameter plus a trailing optional allocator.

**Production instances**: **61 binding emissions** matching the literal pattern `optional_override([](std::initializer_list<...> theInitList, std::optional<occ::handle<NCollection_BaseAllocator>> theAllocator)`.

**Concrete examples** (from the aggregated bindings):

```cpp
// NCollection_List_handle_Expr_GeneralExpression
.constructor(optional_override([](
    std::initializer_list<opencascade::handle<Expr_GeneralExpression>> theInitList,
    std::optional<occ::handle<NCollection_BaseAllocator>> theAllocator) { … }))

// NCollection_List_handle_BOPDS_PaveBlock
.constructor(optional_override([](
    std::initializer_list<opencascade::handle<BOPDS_PaveBlock>> theInitList,
    std::optional<occ::handle<NCollection_BaseAllocator>> theAllocator) { … }))

// NCollection_List_IntTools_SurfaceRangeSample
.constructor(optional_override([](
    std::initializer_list<IntTools_SurfaceRangeSample> theInitList,
    std::optional<occ::handle<NCollection_BaseAllocator>> theAllocator) { … }))
```

**Problem**: embind has no built-in conversion for `std::initializer_list<T>`. The current emission compiles (the lambda is well-typed C++) but JS callers attempting `new oc.NCollection_List_X([...arrayOfHandles...])` cannot reach this constructor because embind's wire conversion does not know how to lift a JS array to a C++ `std::initializer_list<T>`. The constructor is _registered but unreachable_ — a silent JS-side defect with no compile-time signal.

**Best primitive recommendation**: route through `emscripten::val` with explicit JS-array-to-vector conversion inside the lambda body. The lambda receives `emscripten::val theInitList`, validates it is `Array`, iterates it (`for i in 0..length: vec.push_back(theInitList[i].as<T>())`), then passes a `const std::vector<T>&` (or constructed `NCollection_List<T>`) to the C++ side.

**Risks**:

- Per-element wire conversion cost on construction.
- If `T` is `opencascade::handle<U>`, the per-element `.as<opencascade::handle<U>>()` needs handle-type validation.
- TS declaration must read `T[]` not `std::initializer_list<T>`.

**Why the matrix did not cover this row**: the 37-row matrix was built from the parameter-position shapes of _manually authored_ OCCT methods. NCollection bulk-init constructors are emitted by the NCollection auto-discovery generator (per `learned-runtime.mdc`: "NCollection container types are bound via auto-discovery + a generic `using`-declaration generator"). The auto-discovered shape did not pass through the same review surface that produced the matrix.

**Migration sequencing recommendation**: row 38 is **P1** for fix-up but **P0** for documentation — the policy must record the shape exists in production and is currently silent-broken. Production users of `NCollection_List_*` constructors are not exercising this code path today (they use `.Append` instead), which is why the silent breakage has not surfaced as a smoke-test failure. The fix is mechanical (convert lambda to `emscripten::val` + element iteration) but requires the NCollection auto-discovery generator to recognize `std::initializer_list<T>` parameters and emit the val-conversion lambda instead of the raw `std::initializer_list<T>` lambda.
