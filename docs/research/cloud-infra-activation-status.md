# Cloud-Infra Activation Status: Remaining Manual Steps to Complete the Terraform Migration

Single-source-of-truth audit of the `repos/cloud-infra` HCP Terraform capability — what is already in IaC, what is still being applied by hand, and the operator checklist that must be executed to declare the migration "fully active" for both `staging` (`taucad.dev`) and `prod-us` (`tau.new`).

## Executive Summary

The cloud-infra capability is **functionally code-complete** for the V1 slice (Cloudflare DNS + Netlify site build settings + GitHub repo + Netlify env vars + a presence-only Fly secret audit shape). The Netlify staging site (`taucad-staging`, `id=a006a282-6254-499d-a38c-d089b47359c7`) and production site (`taucad-prod-us`, `id=7077c956-6b41-4056-b589-2b3c837435eb`) both deploy from Terraform-managed configuration, DNS authority for `taucad.dev` has migrated from Namecheap to Cloudflare, and the GitHub repo settings (rulesets, environments, App credentials) are reconciled from the `tau-repo` stack. Two production-blocking remediations landed alongside this audit:

1. `modules/netlify-site` now ships a `terracurl_request.repo_link` resource (devops-rob/terracurl provider) that PATCHes the Netlify site with a `repo` block so `prevent_non_git_prod_deploys = true` no longer 422s on first apply.
2. Both stacks declare `netlify_github_installation_id = 96259086` + `repo_path = "taucad/tau"` + `repo_branch` (`main` for staging, `production` for prod-us) so the GitHub App linkage is part of the reviewable plan rather than dashboard click-ops.

**What remains is exclusively operator work** — vendor APIs that intentionally have no programmatic surface (Netlify PAT minting, OAuth redirect URIs, registrar NS edits, Let's Encrypt issuance windows) plus a small set of one-time HCP variable/secret seeds that must precede the first apply on each stack. **The migration is not "blocked on code"; it is sequenced by human gates.** The checklist in [Remaining Manual Steps](#remaining-manual-steps) below is the canonical list — every prior research doc that catalogued a subset of these steps is now superseded by this audit.

## Scope and Non-Goals

**In scope.** Manual operator gates required to activate `repos/cloud-infra` end-to-end across `staging` + `prod-us`: HCP workspace seeding, Netlify GitHub App install, Cloudflare bootstrap, registrar nameserver delegation, Netlify PAT placement, OAuth redirect-URI verification, TLS provisioning wait windows, observability secret placement, and the human smoke gates that precede a production promotion.

**Out of scope.** Application code changes (already merged); the `prod-eu` regional rollout (a follow-up that re-runs the same activation playbook with `enabled = true` on the existing stack template); OpenTofu fallback (documented in the now-superseded `iac-terraform-strategy.md` Appendix D); the observability instrumentation gaps catalogued in `prod-staging-ui-deployment-status.md` (orthogonal to cloud-infra activation — owned by `production-observability-readiness.md`).

## Methodology

1. **Transcript replay.** Walked the [Netlify 422 Terracurl fix](fb0b5c83-9110-41c0-9855-35566d52f8a1) transcript (the in-flight Cloud Infra V1 remediation) to enumerate every operator gate the plan deliberately marked `pending`.
2. **Prior research extraction.** Mined the now-superseded `iac-terraform-strategy.md`, `netlify-ui-deployment-strategy.md`, and `prod-staging-ui-deployment-status.md` for every documented manual step; deduplicated against the runbook content already authored in `repos/cloud-infra/docs/{netlify-iac-runbook.md,dns-migration-plan.md}` and `docs/architecture/production-gitops-runbook.md`.
3. **State verification.** Cross-checked the current state of the Terraform stacks (`repos/cloud-infra/stacks/cloud/{staging,prod-us}`), the modules (`modules/{cloudflare-dns,netlify-site}`), the bootstrap script (`scripts/netlify-bootstrap-site.sh`), and the canonical runbook (`docs/netlify-iac-runbook.md`) to confirm each catalogued "manual step" is still required (vs already absorbed into a Terraform resource).
4. **Categorisation.** Classified residuals into four classes — **first-apply gates** (must complete before the first `terraform apply` succeeds on a given stack), **DNS cutover gates** (registrar NS delegation + TLS wait), **vendor-API-bound gates** (intentionally interactive: PAT mint, OAuth URI), and **promotion gates** (smoke + human judgement).
