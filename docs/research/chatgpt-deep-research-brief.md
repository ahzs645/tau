# ChatGPT Deep Research Brief — Tau Repository

A pre-built deep-research prompt that orients an external researcher (ChatGPT in deep-research mode, or any reader unfamiliar with the codebase) to the Tau monorepo. The block below is **self-contained**: copy everything between the fences and paste it into ChatGPT.

## How to use

1. Copy the entire contents of the fenced block under "Pasteable Brief" below.
2. Paste it into ChatGPT (deep-research / web-enabled mode).
3. Append your specific research question at the end (e.g. _"Compare Tau's filesystem architecture to VS Code's"_ or _"Audit the AI agent middleware stack for context-window robustness"_).
4. ChatGPT will then have a complete map of the Tau repo plus pointers to read external sources (docs, GitHub, blogs, papers).

## Methodology

This brief was assembled by mining the live Tau repo in `/Users/rifont/git/tau` with six parallel readonly exploration subagents covering:

1. Multi-kernel CAD runtime (`packages/runtime`, kernels, transcoders, transport).
2. AI agent system (`apps/api/app/api/chat`, `apps/api/app/api/tools`, `libs/chat`, chat UI).
3. Web UI (`apps/ui`, React Router routes, XState machines, Dockview, Three.js viewer).
4. Filesystem + shared memory (`packages/filesystem`, `packages/memory`, runtime FS bridge, COEP/SAB).
5. API backend (NestJS, Fastify, Drizzle, Better Auth, Socket.IO, OTEL telemetry, Fly/Netlify deploy).
6. Meta layer (`docs/policy`, `docs/research`, `.cursor/skills`, `.cursor/rules`, `tools/`, `scripts/`, `repos.yaml`, `infra/`).

Findings were merged, de-duplicated, and reformatted into a single navigable directory pointer with one-paragraph summaries and 1-3 anticipated questions per area.

## 0. What Tau is

- **Product**: Open-source, MIT-licensed (GPL when bundling OpenSCAD), browser-based CAD platform at https://tau.new.
- **Pitch**: Code-based, parametric, multi-kernel CAD with an AI agent that can read/edit project files, render geometry, and validate output.
- **Repo style**: **Nx monorepo** with **pnpm workspaces**. Apps under `apps/`, runtime libraries under `packages/`, internal libraries under `libs/`, docs and policy under `docs/`.
- **Stack**:
  - **Frontend**: React Router v7, React 19, TypeScript, Tailwind CSS v4, Fumadocs (docs site), Dockview (panels), Allotment (resizable splits), Monaco (editor), Three.js + React Three Fiber (3D viewer), TipTap (chat composer), XState v5 (state machines).
  - **Backend**: NestJS on Fastify, PostgreSQL via Drizzle ORM, Redis (Socket.IO Redis Streams adapter), Better Auth.
  - **AI**: LangChain `createAgent` with LangGraph-compatible Postgres checkpointing; multi-provider (OpenAI, Anthropic, Vertex AI, Ollama, Together AI, Cerebras); Tavily web search; Morph for context compaction.
  - **CAD kernels**: Replicad/OpenCASCADE.js, JSCAD, Manifold, OpenSCAD, Zoo (KCL), and a small "Tau" conversion kernel — all behind a unified `@taucad/runtime` plugin system.
  - **3D processing**: glTF-Transform, Assimp (forked as `@taucad/assimpjs`), lib3mf, tinyusdz (forked).
  - **Observability**: OpenTelemetry (Prometheus + OTLP), Grafana dashboards-as-code, local LGTM stack via `grafana/otel-lgtm` Docker image.
  - **Deploy**: API → Fly.io (`tau-api`); UI → Netlify with branch previews.
- **Cross-cutting design constraints**:
  - **Cross-Origin Isolation (COEP `require-corp` + COOP `same-origin`)** is mandatory in production so that `SharedArrayBuffer` and high-precision timers work for the CAD pipeline. All third-party assets are same-origin proxied (`/api/ph` for PostHog, `/api/github-avatar` for GitHub).
  - **Source-only consumption** of `@taucad/runtime` from the UI — no build artifact between the UI and the runtime kernels.
  - **No extension-based binary detection** — content is sniffed via BOM + NUL heuristic on the first 512 bytes.
  - **Two filesystem watch planes** — kernel fast path (dependency-scoped, drives re-render) vs UI tree path (directory-scoped, drives the file tree). They are never merged.

1. **`defineX` author APIs accept Zod only** — `defineKernel`, `defineTranscoder`, `defineMiddleware`. JSON Schema and defaults are resolved internally so authors don't repeat themselves; `z.input` (not `z.infer`) is used for consumer-facing types so `.default()` optionality is preserved.
2. **Schema-driven runtime APIs use generic inference end-to-end** — kernel formats, transcoder edges, `client.export` options. No `Record<string, unknown>` escape hatches when a schema can drive the type.
3. **No backwards compatibility for unreleased/internal APIs** — roll forward with breaking changes; do not add shims for consumers that don't exist yet.
4. **Kernels are self-contained** — no shared cross-kernel schemas; each kernel duplicates schemas to remain independently extractable.
5. **Agent surface is explicit** — RPCs, tool inputs, system prompts require explicit parameters with no silent `?? mainEntryFile` fallbacks. The LLM must commit to the file/target it intends.
6. **`generatePrefixedId(idPrefix.*)` everywhere** — never `crypto.randomUUID()`. Typed prefixes from `@taucad/types/constants`.
7. **Geometry transfer is two-layer** — byte array (`Uint8Array`) and shared memory (`SharedPool`-backed `SharedArrayBuffer`). The dispatcher's `toTransportGeometry()` auto-stores geometry before checking for pooled delivery (store-then-check with inline fallback). `RuntimeClient.resolveGeometry()` encapsulates SAB → ArrayBuffer resolution so consumers are transfer-mode agnostic.
8. **Browser `TextDecoder` rejects `SharedArrayBuffer`-backed views** — copy at consumer boundary required.
9. *`*fromSafeAsync`(not`fromPromise`)\*\* for all UI XState async actors — Strict Mode and abort safety.
10. **Activity grouping in chat** — assistant tool/reasoning parts are folded into Cursor-style collapsible groups; in-flight group remains visually ungrouped until the next part signals end-of-group; binary file/CAD mutations are never grouped.
11. **Avoid type assertion escape hatches** — `as never`, `as unknown as`, unnecessary `as const` on returns are banned.
12. **No regex-based assertions in tests bound for upstream PRs** — they are hacky and indefensible in review; prefer principled assertions on parsed structures.
