# OCJS Optional-Overload Strategic Review — Opus 4.7

Independent strategic peer review of the in-flight `std::optional<T>` migration described in `docs/research/ocjs-optional-overload-poc-coverage-gaps.md` (the parent doc) and `docs/research/ocjs-optional-overload-resolution-blueprint.md` (the migration blueprint). The parent doc's §Strategic Assessment recommends pausing further `libembind` hunks and building "Corpus C" (an `emscripten::val` + explicit discrimination alternative) before committing more engineering to the optional primitive. This review forms its own conclusion first, then compares with the parallel peer review by gpt-5.5.

## Executive Summary

**Verdict: concur with reservation, refined toward per-shape decision matrix.** The parent doc's diagnosis is correct: `std::optional<T>` and "C++ trailing default argument" are two different semantic primitives that have been forced to share an implementation, and every new production OCCT shape (degenerate siblings, multi-overload trailing defaults, cstring composition, null coercion, sub-2b shadowing) is now generating a new libembind hunk with diminishing upstream alignment. The PoC's 96/96 claim was true against its synthetic corpora but false against the OCCT production shapes that have since broken. The migration is **bounded on the known failures but unbounded on the unknown ones** — exactly the strategic risk the parent doc surfaces in §S.6.

Where I would refine the recommendation: **the choice is not "Corpus B vs Corpus C, pick a winner"**, it is **"which primitive matches each C++ source shape's semantic intent"**. The right framing is a per-shape emitter decision matrix in which `std::optional<T>`, `emscripten::val`-discrimination, native embind overload dispatch, RBV envelopes, JS-effective deduplication, and explicit-suffix escape hatches each own a bounded portion of the problem. `std::optional<T>` keeps its rightful home (genuine `Maybe<T>` returns and parameters — T3 surface, R5 shape 4 const-ref-to-temp); `emscripten::val + isUndefined()/isNull()/instanceof` becomes the canonical primitive for trailing C++ default arguments where the semantic is "absence → use the declared C++ default expression" and where the optional-wildcard branch creates dispatch ambiguity (Findings 1, 2 sub-2b, 3, 4). The val-dispatch path is already idiomatic OCJS — 27,787 usages across 5,324 binding files — so adopting it for trailing defaults is **unifying two patterns, not introducing one**.

I dissent from one detail in the parent doc's §S.3 table: the claim that `emscripten::val`-discrimination requires "**0** libembind hunks" is wrong by 2. The arity-pad hunks (Hunks 1 + 2 in `$ensureOverloadTable` and the ctor dispatcher) are needed for the val-dispatch path too, because the JS surface ergonomic — `obj.Build()` vs `obj.Build(progress)` — only works if omitted positional args are padded to `undefined` before they reach the lambda body. What val-dispatch eliminates is Hunks 3 (optional-wildcard in `$getSignature`), 4 (null→nullopt `toWireType`), 5 (cross-arity type-aware fallback), and 6 (concrete-beats-wildcard precedence) — plus the R4 / R6 / T1 bindgen guards. That is still a substantial reduction (~70% of the new hunk surface and 3 of 3 guards), but it is not zero, and Hunk 6 in particular — proposed as the architectural fix for sub-class 2b — is the load-bearing piece of evidence that the optional approach has crossed an architectural line: **inverting the dispatcher's "first-match-wins" contract to make wildcard lose to higher-arity concrete matches is not a refinement of the dispatcher, it is a re-litigation of the core embind dispatch semantics**. That hunk is upstream-unacceptable (parent doc's own §S.5 rates it "very low" merge probability), and the fact that it is needed AT ALL is a stronger signal than the hunk-count trend in §S.1.

## Q1: Concurrence Check

### Verdict: Concur with reservation

I concur with the parent doc's §S.7 verdict that the optional path is producing a steady stream of new failures, each costing a new libembind hunk, and that the right response is to question the primitive choice rather than to keep adding hunks. The specific claims I find well-grounded:

1. **Semantic mismatch is real and load-bearing** (§S.2). `std::optional<T>` models `Maybe<T>`; trailing C++ default args model `default-on-absence`. The wire-protocol contract column in §S.2 is exactly right: in upstream embind, `undefined ↔ nullopt` and `T ↔ optional<T>(T)`; the OCCT semantic is `(absent) ↔ T(<default-expr>)` and `T ↔ T`. Forcing these into the same wire produces NULL-COERCION (Finding 1), R4 collisions, R6 output-param confusion, and T1 multi-optional collisions exactly as the parent doc predicts.

2. **Patch trajectory is unbounded** (§S.1 table, §S.6 last row). The hunk-count progression (13 → 14 → ~15 → ~18+) is the empirical signal, but the strategic signal is stronger: there is no empirical lower bound on the OCCT shapes that don't compose with `std::optional<T>` because the PoC's synthetic corpora did not exercise the shapes that have broken. The blueprint's "RESOLVED 15/15 risks" claim is technically true but practically misleading because the resolution criterion was synthetic-corpus pass, not real-OCCT-shape coverage.

3. **Sub-class 2b (BRepGProp_Face) is a stronger signal than hunk count.** The parent doc treats sub-class 2b as one of seven migration-attributable failures, fixable by Hunk 6 (concrete-beats-wildcard precedence). I would weight it higher: **the proposed Hunk 6 is itself the smoking gun that the architectural line has been crossed**. Inverting the embind dispatcher's "first-match-wins" contract is not a bounded refinement; it is a re-litigation of how libembind's dispatch resolves any ambiguous signature, full-stop. The parent doc's own §S.5 rates Hunk 6's upstream merge probability as "**very low**" with the rationale "Touches the core dispatch contract; existing apps depending on first-match-wins break." That is the same as saying: we have built a feature that can only ship in a custom dispatcher fork forever. For a fork already maintaining one hard-fought divergence (C1 same-arity dispatch), taking on a second one of greater scope is the wrong direction.

4. **The val-discrimination alternative is not new** (§S.3 first code block). The pattern already exists in OCJS — verified at `TCollection_ExtendedString.cpp:5531` and across the catalog's stated 5,324 binding files / 27,787 usages. Extending it from same-arity-type-discrimination to trailing-default-discrimination is a uniform extension of an established idiom, not a new architectural primitive. This is the strongest argument for the val path.

### Where I refine the parent doc

1. **§S.3 arithmetic correction.** The "Libembind hunks needed for trailing defaults" row in the table claims `emscripten::val + discrimination` requires **0 hunks**. This is wrong by 2 — the arity-pad hunks (1 + 2) are needed for either primitive because they implement the JS-surface ergonomic `obj.Build()` ↔ `obj.Build(progress)`. Without them, every trailing-default-omitted call throws `Function 'X' called with an invalid number of arguments`. What val-dispatch eliminates is Hunks 3 / 4 / 5 / 6 plus the R4 / R6 / T1 guards. The honest comparison is 6 → 2 hunks (and 6 of 6 hunks classified as "Medium-high" or higher upstream merge probability in §S.5), not 6 → 0. This does not change the strategic conclusion but it does change the "fork dissolves into upstream" narrative — the fork still owns Hunks 1+2 plus the entire C1 same-arity dispatch machinery.

2. **The "hybrid strategy" in §S.8 step S6 is the actual recommended target, not a fallback.** §S.8 frames S5 (switch wholesale) and S6 (hybrid) as alternative endpoints depending on whether Corpus C is "substantively worse". I dissent: even if Corpus C is performance-equivalent, the hybrid IS the correct destination, because some shapes (T3 returns, genuine `std::optional<T>` parameters) genuinely have `Maybe<T>` semantics and should use `std::optional<T>` regardless of what we do with trailing defaults. The bench's job is to confirm that val-dispatch is acceptable for the trailing-default shape — not to crown a global winner. This re-framing aligns with gpt-5.5's verdict; I converged on it independently from the §S.2 semantic-axis argument.

3. **R8' patch hygiene (Finding 6 extended) is the load-bearing precondition for any strategic decision.** The five duplicate `$getSignature` definitions in `deps/emsdk/upstream/emscripten/src/lib/libembind.js` mean that **the build pipeline cannot currently produce a deterministic libembind artefact**. Without R8' + R17 (reset to pristine before patching), every Corpus C measurement is non-reproducible across machines and across CI runs. This must land before the A/B/C bench even runs, otherwise the bench results are evidence about local hot-edits, not about the v2 patch as committed.

## Q2: Eigenquestion Identification

### The fundamental question(s)

The migration has been treating one question — **"how do we collapse N fan-out registrations into 1?"** — as if it were the eigenquestion. It is not. It is a surface symptom. The real eigenquestion has three layers:

#### Eigenquestion 1 (semantic): What does "absence" mean at this parameter position?

Every parameter position in every OCCT method/ctor must be classified into one of four "absence semantics":

| Absence semantic          | C++ source shape                                               | JS-surface meaning of omission                    | Right primitive                                                   |
| ------------------------- | -------------------------------------------------------------- | ------------------------------------------------- | ----------------------------------------------------------------- |
| **A. Default-on-absence** | `T x = D` (trailing default with concrete default expression)  | "I did not specify; use the C++ declared default" | `val` + `isUndefined()/isNull() → D : arg.as<T>()` in lambda body |
| **B. Maybe-T**            | `std::optional<T>` (genuine source-level optional)             | "I am explicitly conveying no-value"              | `std::optional<T>` natively; `nullopt ↔ undefined`                |
| **C. Output sink**        | `T&` (non-const reference, output param)                       | (omission not valid — output)                     | RBV envelope; strip JS input slot if pure-out                     |
| **D. Polymorphic**        | same-arity overload set with disjoint JS-distinguishable types | "I am providing this specific type variant"       | `val` + `instanceof / typeOf` discrimination in lambda body       |

The current migration conflates A with B by emitting `std::optional<T>` for both. This is the source of every failure in Findings 1, 2 sub-2b, 3, 4 — all are downstream of treating A as if it were B.

Critically, **the answer to "when omitted-argument semantics mean 'use the default', is that the case for ALL optional arguments?"** is **no**. R5 shape 2 (handle defaults with `= Handle()`) is sometimes A (the C++ default is the null handle, which is meaningful and the OCCT caller-pattern), sometimes effectively B (some OCCT signatures treat null handle as "no value provided, do nothing"). The bindgen cannot distinguish these without per-OCCT-method semantic annotation. The safe default is to treat all `=` defaults as A (default-on-absence with the literal C++ default expression), and to reserve B for source-level `std::optional<T>` declarations.

#### Eigenquestion 2 (representability): What distinctions does the JS runtime preserve?

For each parameter position, what discrimination can JS actually express? The catalog:

- **Preserved**: `instanceof` on registered classes; `typeof` for `string`/`number`/`boolean`/`object`/`undefined`; `Number.isInteger`; `null` vs `undefined` (when the dispatcher does not short-circuit); enum-membership via `Module.EnumType.X`; array/typed-array.
- **Not preserved**: `int` vs `size_t` for ordinary JS numbers; `char` vs `const char*` when both surface as strings; const-vs-non-const overloads; SFINAE/template-specialization branches; nominally-different but JS-identical class hierarchies (e.g. shape-base coercion in OCCT).

This is also the answer to "what other C++ semantics must be solved for" — every entry in the right column requires a non-`std::optional`-based primitive:

- Output params (`T&`) → RBV envelope (gpt-5.5's row 14–17)
- Reference passthrough (`T&` const for input) → native typed binding
- Smart-pointer reassignment (handle outputs reassigning the handle, not the underlying T) → RBV input elision (`ocjs-rbv-handle-output-param-elision.md`)
- Runtime type discrimination → `emscripten::val` + dispatch in lambda
- Integer twins (`size_t` vs `int`) → bindgen JS-effective signature dedup (`ocjs-rbv-non-copyable-and-integer-overload-dedup.md`)
- `char` vs `cstring` → TS-only classification + suffix escape; runtime collapses both to string
- Real `std::optional<T>` returns → native `std::optional` (T3 surface)
- Real `std::optional<T>` params → native `std::optional` with explicit-undefined-policy doc
- Defaulted handle/object with non-null default → `val` + `isUndefined() ? <build default> : arg.as<T>()`
- Defaulted scalar policy flags → `val` + `isUndefined() ? D : arg.as<T>()` (same as #1 with T = bool/enum/double)
- RBV-non-copyable returns → RBV envelope with deleted-copy carve-out (`ocjs-rbv-non-copyable-and-integer-overload-dedup.md`)
- Mixed-return overload groups → `_emitValDispatchMethod` with `mixed_returns=true`
- RBV-elided arity collisions → JS-effective arity dedup before optional/val emission
- NCollection template surfaces → source-level template instantiation discovery + downstream typed/val dispatch
- ADL/free functions → explicit generated facades, no overload runtime
- Nullable object args → explicit null-policy per OCCT class
- Explicit-undefined args → covered by absence-semantics classification (eigenquestion 1)
- SFINAE/deleted overloads → filter at source

The point of this enumeration is that **`std::optional<T>` is the right answer for exactly 2 of these (T3 returns, T3 params) and partial-right answer for 1 more (defaulted scalar/object with literal C++ default — but only if dispatch ambiguity does not arise)**. Every other row needs a different primitive. The migration's failure mode is treating `std::optional<T>` as the universal answer.

#### Eigenquestion 3 (ergonomics): What should the JS API look and feel like?

This is gpt-5.5's "JS interface should mimic the C++ interface in intent" framing, which I also arrived at independently — see Q3 below. The key insight: ergonomic mimicry does **not** mean transliterating every C++ parameter literally; it means preserving the **intent** of the C++ surface. `surface.Bounds()` returning `{U1, U2, V1, V2}` mimics the intent of `Bounds(double&, double&, double&, double&)` even though it does not mimic the syntax. This is already what RBV does in OCJS. Trailing defaults should follow the same intent-preservation rule: `obj.Build()` mimics the intent of `obj.Build(<default-progress>)`; the lambda body materializes the default. This is achievable equally with `std::optional<T>` and with `emscripten::val` — the choice between them is about dispatch ambiguity, not ergonomics.

### Single most important question (per gpt-5.5's framing, restated independently)

**For each C++ parameter position, what semantic distinction must be preserved, and does the JavaScript runtime have a representation that preserves it without ambiguity with adjacent overloads' parameter positions?**

The "without ambiguity with adjacent overloads' parameter positions" qualifier is the key insight from the BRepGProp_Face regression: an `optional<bool>` slot in isolation is fine; an `optional<bool>` slot at arity-N that aliases an arity-(N+1) `(T1, optional<bool>)` slot's first position is fatal under the optional-wildcard dispatch rule. The eigenquestion must include the inter-overload context, not just the per-position context.

## Q3: Reframing Test

### Framing: "The JS interface should mimic the C++ interface, both ergonomically and in intent"

I converged on this framing independently from the §S.2 semantic-axis argument, and it makes the problem tractable. The framing partitions the implementation choices cleanly:

1. **Ergonomic mimicry** means the JS caller's call-site looks like the C++ caller's — `obj.Build()`, `obj.Build(progress)`, `new oc.gp_Pnt(1, 2, 3)`, `surface.Bounds()`. This is a JS-surface contract, not a wire-protocol contract.
2. **Intent mimicry** means the JS API preserves the C++ source's semantic meaning — `SetColor(label, color)` vs `SetColor(shape, color)` remain two callable variants of `SetColor` (not `SetColor_1` / `SetColor_2`); `surface.Bounds()` returns the bounds object (not "give me an out-param ceremony"); `chamfer.Build()` runs the build with the default progress (not "tell me what optional argument you want me to pass").

Ergonomic + intent mimicry **does not dictate a wire-protocol primitive**. It is a requirements specification. The wire protocol must satisfy it; how it does so is the implementation question. The right primitive for each parameter position is the one that satisfies ergonomic + intent mimicry **with the lowest dispatch ambiguity, the smallest libembind delta, and the strictest upstream alignment**.

### The right reframing answers the question "is the framing right?" with: yes, with a caveat

The framing is right BUT it must be applied with the additional constraint that **the dispatch ambiguity caused by an emission must be evaluated against the full overload set of the surrounding class, not just per-method-position**. The sub-class 2b regression is the worked example: the `optional<bool>` choice would be locally correct for the smaller ctor in isolation, but in the context of the larger ctor with a `TopoDS_Face` first param it creates a wildcard short-circuit. The framing is necessary but not sufficient; the additional context-awareness rule is what would have caught BRepGProp_Face at PoC time.

### Rows gpt-5.5 missed (or under-emphasized)

gpt-5.5's matrix is 30 rows covering most of the same ground; the rows I added independently:

- **Row 5 (function-call-expression default)** — gpt-5.5 collapsed this into row 1; I separated because the per-call cost calculus differs (val's lazy-default vs optional's eager `NS::Func()` evaluation).
- **Row 23 (defaulted handle with non-null default)** — gpt-5.5's row 21 covers defaulted handles but does not flag the silent-corruption mode where `value_or(Handle())` overrides a non-null C++ sentinel. This is a NULL-COERCION sibling that has not yet been seen in production but is latent.
- **Row 27 (RBV-elided arity collision composed with optional)** — gpt-5.5's row 25 covers it but I would strengthen the recommendation: the bindgen must compute JS-effective arity AFTER RBV elision AND AFTER default expansion BEFORE checking for dispatch collisions. This is the missing step that hides the BRepGProp_Face-class of regressions.
- **Row 33 (TR-CW cstring + trailing default with val)** — gpt-5.5 does not have a cstring-specific row; I added it because the existing cstring-wrapper lambda composes with val more naturally than with optional (no `value_or(...).c_str()` nesting, no null-coercion hunk needed).
- **Row 34 (TR-MO multi-overload + trailing default)** — gpt-5.5's row 4 covers the general "multi-overload one sibling has trailing defaults that overlap another sibling's arity" but does not call out that val + optional are functionally equivalent for this row.
- **Row 36 (`T{}` default-constructed)** — minor refinement of row 1/24.
- **Row 37 (reference-default to singleton)** — covers a R6-adjacent shape that the existing R6 guard catches but the matrix should still document.

### Rows where I converged independently with gpt-5.5

The matrix's rows 6, 9–14, 16–22, 26, 28, 29, 31, 32 all match gpt-5.5's analysis substantively (numbering differs). This is the strongest signal of independent convergence — when two reviewers reading the same source independently arrive at the same per-shape verdict, the verdict is robust.

## Final Recommendation

**Adopt the hybrid (matrix-driven per-shape) target as the migration's North Star, not as a fallback or alternative.** Specifically:

1. **Immediately (P0, before any further migration work):**
   - Land AO6 (patch-application hygiene reset-to-pristine) and AO7 (Path B as canonical hunk). These are preconditions for any reproducible measurement.
   - Land AO1 (`docs/policy/ocjs-trailing-default-emission-policy.md`) with the §Q4 matrix as the source-of-truth.
   - Land AO2 (context-aware overload-aliasing detector in bindgen) — this is the architectural fix for sub-class 2b that obviates the need for Hunk 6.
   - **Stop adding** Hunks 4, 5, 6 to the v2 patch. Keep Hunks 1, 2, 3, and add Path B as Hunk 4-canonical.

2. **Next (P0, the actual Corpus C work):**
   - Build Corpus C covering all 37 matrix rows (NO1–NO9 + R9–R14 from the parent doc).
   - Score per-row per AO4 / MO3. Output is matrix verdict refinements, not a single winner.
   - Validate AO5 (JS-effective arity precondition) lands as a bindgen emit-time check.

3. **Then (P1, the actual migration):**
   - Per the refined matrix, emit `std::optional<T>` for rows {3, 4, 5, 21, 22} (and conditionally for rows {1, 24, 36} where AO2 confirms no sibling aliasing).
   - Emit `emscripten::val` discrimination for all other default-bearing rows.
   - Land AO9 (CI guard against future dispatcher-precedence-inversion hunks).
   - Update the migration blueprint's Acceptance Criteria per MO6 / MO4 / DO4.

4. **Long-term (P2):**
   - The C2 portion of the libembind patch (the optional-wildcard hunk + register_optional<T> machinery) becomes a stable, bounded fork commitment for rows {3, 4, 5, 21, 22} only.
   - The C1 portion (same-arity type dispatch, suffix-free overloads) remains the load-bearing fork commitment per the prior strategic-direction assessment.
   - Future emcc upgrades require validating only Hunks 1, 2, 3 + Path B + the bounded register_optional usage — not the unbounded Hunks 4, 5, 6 trajectory the current path implies.

The migration's headline goal — collapse N fan-out registrations into 1 per method — is achievable under the hybrid. The semantic correctness goal — preserve the C++ source's intent on the JS wire — is **only** achievable under the hybrid. Single-primitive paths sacrifice one for the other.
