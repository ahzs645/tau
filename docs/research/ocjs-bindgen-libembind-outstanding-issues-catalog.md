# OCJS Bindgen + libembind Outstanding Issues Catalog

Single source of truth for every unresolved OCJS bindgen/libembind defect, the deterministic smoke test that asserts it, and the post-gen hot-patches that still need to be promoted to permanent fixes in `bindings.py` / `libembind-overloading.patch`.

## Executive Summary

The two-rebuild validation pass on `repos/opencascade.js` (libembind R1+R2 `Object.hasOwn` gates + bindgen `is_override` guard removal) reduced the smoke matrix from 69 failed to 12 failed (out of 400). This document inventories every outstanding bindgen/libembind issue, anchors each to a smoke test (existing or to-be-added), and lists the runtime-only patches that still need a permanent upstream fix.

**Strategic framing** (see [`docs/research/ocjs-libembind-strategic-direction-assessment.md`](ocjs-libembind-strategic-direction-assessment.md) for the full analysis): the 8 defects split into two architecturally distinct clusters, **C1** (same-arity type dispatch — load-bearing for v3's suffix-free overloads, no upstream-canonical alternative) and **C2** (arity fan-out for trailing C++ defaults — upstream-canonical solution exists via `std::optional<T>` + Emscripten 3.1.68's PR #22591). 5 of the 8 catalog defects (FO-R3, TR-CW, TR-MO, TR-RBV, TR-GATE) are C2-cluster and **collapse to zero work** if bindgen retires arity-fan-out in favour of `std::optional<T>` emission. The 3 residual defects (CTJ-1/CTJ-2, FO-R1, DF-R2b) are C1-cluster and need architectural fixes regardless.

**Headline counts**:

- **8 outstanding source-level defects** across bindgen and libembind, of which **5 are dead-end work** if the strategic-direction assessment's Option C (bifurcate, retire C2) is adopted.
- **3 architectural defects in the C1 cluster** (CTJ-1+CTJ-2, FO-R1, DF-R2b) that need fixes regardless of strategic direction.
- **4 deterministic regression pins added** in this catalog-completion pass (3 smoke, 1 parity) covering TR-CW, TR-MO, TR-RBV, TR-GATE; CTJ-1 + CTJ-2 are anchored by the pre-existing behavioural smoke tests `smoke-cstring-dispatch.test.ts` + `smoke-ambiguous-overloads.test.ts` (any val-dispatched constructor catastrophically fails without Path B). All nine defect IDs now have anchor tests.
- **2 hot-patches on compiled output** that need to be promoted into the canonical patch file or bindgen.

## Problem Statement

After the most recent two-rebuild validation pass, the OCJS WASM build still ships with:

1. Twelve real smoke-test failures across four test files spanning two failure modes (cross-sibling dispatch in real multi-level OCCT inheritance; signature-array entries for `Handle<>`-elision overloads).
2. One critical workaround (`cppTypeToJsType` minifier elimination) that is **only** applied as a sed/python edit to the compiled `build-configs/opencascade_full.js` artifact. Every clean rebuild re-introduces the bug and the workaround has to be re-applied manually.
3. Several known bindgen gaps catalogued in [`repos/opencascade.js/TODO.md`](../../repos/opencascade.js/TODO.md) that have no deterministic smoke coverage and therefore cannot be measured for progress.

Without a single canonical inventory, progress between rebuild cycles is hard to measure and easy to regress. This document fixes that.

## Methodology

1. Re-read [`docs/research/ocjs-trailing-default-arity-fan-out.md`](ocjs-trailing-default-arity-fan-out.md), [`docs/research/ocjs-embind-js-dispatch-failures.md`](ocjs-embind-js-dispatch-failures.md), and [`docs/research/ocjs-thrusections-build-arg-trap.md`](ocjs-thrusections-build-arg-trap.md) to recover the FO-R*, DF-R*, TR-R\* catalogues.
2. Walked the reconciliation report from the most recent rebuild cycle (`/tmp/ocjs-smoke-rebuild2.log`) and identified every failing test plus its anchor bug.
3. Audited [`repos/opencascade.js/src/patches/libembind-overloading.patch`](../../repos/opencascade.js/src/patches/libembind-overloading.patch) and [`repos/opencascade.js/src/ocjs_bindgen/codegen/bindings.py`](../../repos/opencascade.js/src/ocjs_bindgen/codegen/bindings.py) to confirm which fixes are present in source-of-truth vs. only in the compiled artefact.
4. Diff'd the compiled `build-configs/opencascade_full.js` against what a clean rebuild emits to identify residual post-gen mutations.
5. Cross-referenced [`repos/opencascade.js/TODO.md`](../../repos/opencascade.js/TODO.md) to surface latent bindgen items the audit must include.

CTJ-1 + CTJ-2 are deliberately **not** pinned by a dedicated patch-integrity test. An earlier `tests/patch-integrity/libembind-patch-markers.test.ts` was drafted but dropped before completion because it asserted on minified-JS string literals and patch-source comments — both are internal implementation details, not observable OCJS behaviour. Their observable surface (every val-dispatched constructor) is already pinned catastrophically by `tests/smoke/smoke-cstring-dispatch.test.ts` (7 tests across `TCollection_AsciiString` + `TCollection_ExtendedString` ctor overloads) and `tests/smoke/smoke-ambiguous-overloads.test.ts` (TCollection section). A regression in Path B would also light up 13+ downstream files that depend on `TCollection_AsciiString("…")` (STEP, IGES, XCAF, glTF, OBJ, PLY, BRepGraph, expressions, document framework, file I/O). CTJ-1 has no behavioural surface independent of CTJ-2; Path B IS the runtime mitigation.

Coverage budget held: each new test is < 80 lines, mirrors the `describe.skipIf(!wasmExists)` + `using` RAII pattern used across the existing smoke suite, and pins a single defect ID. Composite tests were explicitly avoided — one defect per file makes the failure log self-explanatory.

### Finding 3: Post-gen patches lacking permanent fixes

The latest build artefacts contain two surgical edits to `build-configs/opencascade_full.js` that are **not** present in `src/patches/libembind-overloading.patch`. Every clean rebuild re-introduces the underlying defect and the workaround has to be re-applied by hand.

| ---------------------------------- | ------------------------ | ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `Cc` (cppTypeToJsType) restoration | `Cc=a=>{var b=R[a];…}`   | CTJ-1             | Three source-level rewrites in `libembind-overloading.patch` (`if/else if` chain → flat `if (cond) return X;` → top-level `cppTypeToJsTypeNameTable` lookup) all produced byte-identical compiled output. The `emscripten::val` and `valueType` branches are stripped regardless of source shape. Suspect: emcc's `acorn-optimizer` AJSDCE pass or babel preset-env doing reachability analysis based on observed call-sites at build time. | Two viable directions: (a) move the dispatch out of `libembind.js` entirely into a `__postset` JS that runs after library binding (the optimizer cannot reason across postset boundaries); (b) annotate the function with an emcc-recognised "do not optimize" hint (need to verify such a hint exists for library JS). Path B is a defence-in-depth layer that only matters if `Cc` continues to be unreliable.                                                                                                                                                                                                      |

Until the permanent fixes land, the build workflow has a hidden manual step: `OCJS_HOTPATCH=1 ./scripts/apply-cpptypetojstype-hotpatch.py build-configs/opencascade_full.js` (script to create — currently this is done by hand). Codifying the hot-patch as a build-step script is the **minimum** acceptable interim measure so it stops being a tribal-knowledge gotcha; the goal remains a true source-level fix.

## Appendix B: user-facing impact on replicad and direct OCJS consumers

### B.1 TL;DR

- **FO-R1 (real-OCCT cross-sibling) is build-config dependent, NOT a universal production blocker.** The cross-sibling stomp fires only when the build YAML registers a symbol whose registration mutates an ancestor's inherited overload table (most notably `BRepFeat_SplitShape`). Replicad's 227-symbol ship YAML excludes such symbols — the trigger never registers, the bug never fires, and replicad's 801/801 kernel tests + 8 example models (including the birdhouse rendered in the Tau UI today) all pass. The OCJS smoke suite tests against the 4,441-symbol `full.yml` which DOES include `BRepFeat_SplitShape`, hence the test failures. **35 of replicad's 227 bound classes inherit from `BRepBuilderAPI_Command`; the inheritance graph alone does not predict whether FO-R1 fires — the corrupting siblings' presence in the YAML does.** See the [strategic-direction assessment](ocjs-libembind-strategic-direction-assessment.md) Finding 1 for the reproduction evidence.
- **CTJ-1 / CTJ-2** would have catastrophic impact (STEP export, assembly export, every string-constructed OCCT type) but replicad's current `replicad_single.js` artefact already ships the **Path B getSignature hot-patch**, so user-facing impact for replicad consumers is fully mitigated as long as the WASM artefact is rebuilt against the same patched libembind.
- **DF-R2b** does not impact replicad — replicad never calls `BRep_Tool.PolygonOnTriangulation` / `PolygonOnSurface`.
- **FO-R3** (arity-0 inherited `Build()`) does not impact replicad — replicad always passes `progress` to `.Build(progress)`. **And FO-R3 disappears entirely under the strategic-direction assessment's Option C** (no truncation lambda needed if defaults are `std::optional<T>`).
- **TR-CW / TR-MO / TR-RBV / TR-GATE** are bindgen completeness gaps that **all dissolve under Option C**. Their smoke pins (Finding 2) document the defects, but under Option C the pins themselves get retired because the underlying gates do not exist.

### B.3 What this means in practice

For replicad consumers using `@taucad/replicad-opencascadejs` from the current branch, the hot-patched `replicad_single.js` artefact already in `repos/replicad/packages/replicad-opencascadejs/src/` is **fully functional**:

- ✅ STEP export, assembly export, color/material attribution work (CTJ-1/CTJ-2 fully mitigated by Path B).
- ✅ Simple primitives, individual sweeps, individual booleans work in isolation.
- ✅ Models mixing boolean ops + fillets/chamfers work (verified: 801/801 kernel tests pass, 8 example models including birdhouse render correctly in the Tau UI). FO-R1 does NOT fire because replicad's YAML excludes the corrupting sibling registrations (notably `BRepFeat_SplitShape`).

The smoke evidence and the production evidence are consistent: `smoke-fillets-chamfers.test.ts` fails in the **full.yml** build because that build includes `BRepFeat_SplitShape`. The same dispatcher in replicad's 227-symbol build succeeds because the corrupting sibling was never registered. The fix is still architecturally important (any consumer custom-trimming a YAML that includes a corrupting sibling will hit FO-R1 without warning), but the prior framing of FO-R1 as a universal production blocker for "virtually every real CAD model" was incorrect — corrected per the [strategic-direction assessment](ocjs-libembind-strategic-direction-assessment.md) Finding 1.

### B.4 Replicad fragility against the hot-patch lifecycle

The most important consequence of this analysis: **replicad is silently fragile against the hot-patch lifecycle for CTJ-1/CTJ-2**, NOT against FO-R1. The Path B fallback in `getSignature` is what keeps replicad's STEP/assembly export working — if a future rebuild loses Path B (e.g. patch hunk-header drift, or a minifier change that strips it the way it strips `cppTypeToJsType`), replicad's exporters will fail catastrophically without warning. The R2/R3 recommendations directly address this lifecycle fragility. FO-R1 is a separate concern, build-config dependent, and not currently triggered by replicad's symbol selection.

### B.5 What is NOT broken in replicad

- All `BRep_Tool` usage (no `Polygon*` calls)
- All arity-0 `Build()` patterns (none on Command-derived classes)
- All `Transfer()` calls (explicit 4-arg form)
- All primitive constructors with non-ambiguous typed args
