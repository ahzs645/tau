# Safari SVG Rendering Compatibility Audit

Investigation into two distinct Safari rendering failures of bespoke SVG assets in `apps/ui`: (1) the `LogoLoader` flex animation freezes on its starting frame, and (2) several icons rendered through `<SvgIcon>` lose their gradients, masks, or drop-shadow filters because the sprite is referenced as an external document.

## Executive Summary

Two unrelated WebKit limitations are causing the regressions:

1. **LogoLoader freeze** — `LogoLoader` morphs paths by animating the SVG `d` attribute via CSS `@keyframes`. WebKit has never shipped CSS animation/transition support for the SVG 2 `d` Geometry Property; Safari therefore renders only the first keyframe state and never advances. Chrome 60+ and Firefox 97+ ship the feature, which is why the animation works there.
2. **Icons missing in sprite** — `<SvgIcon>` references icons through an external sprite (`<use href="sprite.svg#id">`). When a `<symbol>` in an externally referenced sprite carries `<filter>`, `<mask>`, `<clipPath>`, or `<linearGradient>` definitions, Safari (and to a lesser extent other engines) silently drops those `<defs>` from the instantiated shadow tree. The `opencascadejs` icon is the most visible casualty because every visual layer depends on a `<filter>` and a `<mask>`; gradient-based brand icons (`autodesk`, `cursor`, `gemini`, `meta`, `qwen`, `bedrock`) are partially affected.

The lowest-risk fix is **(a)** swap the CSS keyframes in `LogoLoader` for a SMIL `<animate attributeName="d" …>` (universally supported, including Safari), and **(b)** inline the generated sprite once at the app shell using a `?raw` import + `dangerouslySetInnerHTML`, so every `<use href="#id">` becomes a same-document reference. Both changes are localised, do not regress Chromium/Firefox, and unblock every existing icon without rewriting source SVGs.

## Problem Statement

Two user-reported regressions, both Safari-only:

1. The CAD app's connecting screen renders the Tau "Connecting…" pill with the brand mark frozen in its closed state. Chrome animates the three flexing path segments smoothly. The pill itself, the text, and the surrounding layout are pixel-identical between browsers — only the SVG morph is missing.
2. The kernel selector renders most brand icons correctly, but the `OpenCascade` button renders an incomplete glyph (the masked white outline ring is missing and the gradient circle is reduced to a flat dark fill). Other gradient-heavy logos (Autodesk, Meta, Cursor, Gemini, Qwen, Bedrock) lose their fills entirely or render as solid black silhouettes depending on viewport size and zoom.

The two symptoms share a common surface (`<svg>` content in `apps/ui/app/components/icons` and `apps/ui/app/components/ui/loader.tsx`) but stem from independent WebKit limitations.

## Methodology

1. Read both call sites: `apps/ui/app/components/ui/loader.tsx`, `apps/ui/app/components/logo-loader.tsx`, and `apps/ui/app/components/icons/svg-icon.tsx`.
2. Inspected the raw source SVG for `opencascadejs` (`apps/ui/app/components/icons/raw/opencascadejs.svg`) and confirmed the structure (`<filter>` + `<mask>` + 2× `<linearGradient>`).
3. Inventoried every raw icon for `<filter>`, `<mask>`, `<linearGradient>`, `<radialGradient>`, and `<clipPath>` to find every icon at risk.
4. Inspected the generated sprite (`apps/ui/app/components/icons/generated/sprite.svg`) to confirm the per-symbol IDs are properly namespaced (so the bug is **not** an ID collision).
5. Cross-checked behaviour against published browser-compatibility data (`SVGator` matrix, MDN, WebKit bug 11543103, Stack Overflow Q&A from `herrstrietzel` documenting the cross-document `<use>` `<defs>` hand-off).
6. Reviewed `apps/ui/vite.config.ts` to confirm how the sprite is bundled (Vite `?url` default for `.svg` imports, generated by `vite-svg-sprite-wrapper` with `enableSpriteGeneration` toggled per-build).

## Findings

### Finding 1: `LogoLoader` animates the SVG `d` attribute via CSS, which Safari does not implement

`LogoLoader` defines three keyframes that morph the `d` attribute of three `<path>` elements:

```5:18:apps/ui/app/components/logo-loader.tsx
  @keyframes tau-flex-left {
    0%, 5% { d: path("M14.17 87.42 L62.54 99.9 L239.88 145.69 L239.88 462.05 L239.88 512 L191.51 499.51 L191.51 183.16 L14.17 137.37 Z"); }
    20%, 46% { d: path("M14.17 87.42 L62.54 99.9 L62.54 416.26 L239.88 462.05 L239.88 512 L191.51 499.51 L14.17 453.73 L14.17 137.37 Z"); }
    61%, 100% { d: path("M14.17 87.42 L62.54 99.9 L239.88 145.69 L239.88 462.05 L239.88 512 L191.51 499.51 L191.51 183.16 L14.17 137.37 Z"); }
  }
  @keyframes tau-flex-top {
    0%, 12% { d: path("M256 66.6 L384.97 33.3 L481.7 58.28 L384.97 83.25 L256 116.55 L127.03 83.25 L30.3 58.28 L127.03 33.3 Z"); }
    27%, 53% { d: path("M256 0 L384.97 33.3 L481.7 58.28 L384.97 83.25 L256 49.95 L127.03 83.25 L30.3 58.28 L127.03 33.3 Z"); }
    68%, 100% { d: path("M256 66.6 L384.97 33.3 L481.7 58.28 L384.97 83.25 L256 116.55 L127.03 83.25 L30.3 58.28 L127.03 33.3 Z"); }
  }
```

The `d` property is defined by the SVG 2 spec as a CSS Geometry Property (https://svgwg.org/svg2-draft/paths.html#TheDProperty). Implementation status across browsers (current as of April 2026):

| Engine                    | CSS-animatable `d`? | First version     | Notes                                                 |
| ------------------------- | ------------------- | ----------------- | ----------------------------------------------------- |
| Blink (Chrome/Edge)       | ✅                  | Chrome 60         | Implementation pre-dates the final spec text          |
| Gecko (Firefox)           | ✅                  | Firefox 97 (2022) |                                                       |
| WebKit (Safari macOS/iOS) | ❌                  | Never             | Tracked as WebKit bug 11543103, no expressed timeline |

The SVGator browser-support matrix lists "Morph (d attr.)" as ❌ for Safari and ✅ for every other modern engine. The bug is not in our keyframes — Safari simply ignores the property and keeps each path locked to whatever `d` attribute was set on the static element (line 28, 33, 38). Because the static `d` on each path matches the `0%` keyframe, Safari freezes on what looks like the loader's "rest" pose.

This is the same root cause as the Stack Overflow canonical "CSS d path attribute doesn't work in Safari, FireFox" question (`stackoverflow.com/questions/46454102`) and is reaffirmed in 2025 answers (`stackoverflow.com/questions/79434636`).

### Finding 2: Sprite `<use>` references break filters/masks in WebKit because the sprite is loaded as an external document

`<SvgIcon>` resolves icons through:

```62:69:apps/ui/app/components/icons/svg-icon.tsx
  const aliasTarget = (iconAliases as Record<string, SvgIcons | undefined>)[id];
  const resolvedIconId: SvgIcons = aliasTarget ?? (id as SvgIcons);

  return (
    <svg {...properties} className={className} viewBox='0 0 56 56'>
      <use href={`${spriteSvg}#${resolvedIconId}`} />
    </svg>
  );
```

`spriteSvg` comes from `import spriteSvg from '#components/icons/generated/sprite.svg'`, which Vite resolves to an asset URL (`/assets/sprite-<hash>.svg`). Every `<use>` element therefore references **a separate document** loaded via fragment identifier — exactly the configuration in which all browsers (and Safari most aggressively) restrict which `<defs>` they will materialise inside the cloned shadow tree.

The behaviour, confirmed across multiple Stack Overflow root-cause investigations from `herrstrietzel`, is:

| Definition inside external `<symbol>`   | Chromium                          | Firefox                 | Safari                                                                                  |
| --------------------------------------- | --------------------------------- | ----------------------- | --------------------------------------------------------------------------------------- |
| `<linearGradient>` / `<radialGradient>` | ✅ rendered                       | ✅ rendered             | ⚠️ inconsistent — frequently dropped on first paint, restored after layout perturbation |
| `<filter>` (drop-shadow, Gaussian blur) | ✅ applied                        | ⚠️ applied with caveats | ❌ frequently dropped; visible artefacts even when applied                              |
| `<mask>`                                | ⚠️ omitted in many configurations | ✅ applied              | ❌ silently omitted                                                                     |
| `<clipPath>`                            | ⚠️ omitted                        | ⚠️ omitted              | ❌ silently omitted                                                                     |

When the definition is dropped, the affected element falls back to its base attributes — so a path that was supposed to be filled with `url(#paint0_linear)` renders as the default `fill="#000"`, and a `<g filter="url(#filter0_d)">` renders without its drop shadow. The visible regression in the kernel selector matches this pattern exactly.

### Finding 4: Sprite generator already namespaces IDs — collision is not the cause

`vite-svg-sprite-wrapper` runs SVGO with the default `cleanupIds` plugin, so the generated sprite uses short suffix-style IDs (`aaa`, `bea`, `bec`, …) that are unique across symbols:

```
opencascadejs: ids=['opencascadejs', 'bea', 'beb', 'bec', 'bed']  refs=['bea', 'beb', 'bec', 'bed']
autodesk:      ids=['autodesk', 'acc', 'acb', 'aca', 'ace', 'acd', 'acf']
meta:          ids=['meta', 'aza', 'azb', 'azc', 'azd', 'aze', 'azf', 'azg', …]
3ds-max:       ids=['3ds-max', 'aaa']  refs=['aaa']
```

References stay scoped within the same `<symbol>`, so the bug is not "two icons with the same gradient ID collide". The bug is purely the cross-document hand-off of `<defs>` content described in Finding 2.

### Finding 5: PNG escape hatch already exists, indicating the team has hit similar issues before

`<SvgIcon>` already has a fallback for `manifold` that renders the icon as a `<image>` element pointing at a PNG:

```7:9:apps/ui/app/components/icons/svg-icon.tsx
const pngIcons = {
  manifold: manifoldPng,
} as const satisfies Record<string, string>;
```

This pre-existing escape hatch confirms that complex SVGs in the sprite have caused rendering issues before. The pattern works but loses crispness at high-DPI/zoom and increases the bundle by raster size; it is acceptable for one-off cases but not a systemic solution.

### Finding 6: Sprite generation is opt-in, so source-only changes need a sprite rebuild

`apps/ui/vite.config.ts` ships with `enableSpriteGeneration = false`. Anyone editing a raw SVG must flip the flag, run a build, and commit `sprite.svg` + `svg-icons.d.ts`. This is fine for the sprite contents, but it also means a "fix the SVG, regenerate" loop is not enough on its own — the underlying cross-document limitation will keep biting until we change how the sprite is consumed.

## Trade-offs

### LogoLoader fix options

| Option                                                               | Cross-browser  | Notes                                                                                            |
| -------------------------------------------------------------------- | -------------- | ------------------------------------------------------------------------------------------------ |
| **SMIL `<animate>`** (recommended)                                   | ✅ all engines | Native SVG animation; declarative; same path data; CSS `cubic-bezier` replaced with `keySplines` |
| `requestAnimationFrame` JS morph                                     | ✅ all engines | Adds runtime cost; prevents prerender; harder to read                                            |
| Replace path morph with CSS `transform: scale` of three static paths | ✅ all engines | Loses the squash/stretch silhouette that makes the brand mark feel alive; visually different     |
| Lottie / animated GIF                                                | ✅ all engines | Bigger payload; no `currentColor` support; loses theme-awareness                                 |

SMIL is the lowest-cost option and preserves the existing visual exactly (same `d` strings, same timing). SMIL was deprecated in Chromium briefly (2015) but the deprecation was reversed, and it remains the canonical cross-browser SVG animation primitive recommended by the WHATWG SVG WG.

### Icon-sprite delivery options

| Option                                                                         | Pros                                                                                                                                                                        | Cons                                                                                                                                                              |
| ------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Inline sprite once at app shell** (recommended)                              | All `<use>` references become same-document; gradients/filters/masks all work in every browser; single network request still cached aggressively; no per-icon source change | ~100 KB inline payload at app shell instead of separately cached file (sprite is currently 103 727 bytes); blocks initial HTML by that amount unless we stream it |
| Per-icon React components (`?react` Vite plugin or hand-rolled like `tau.tsx`) | Tree-shakable; no sprite at all                                                                                                                                             | Loses the single-asset cache; +43 component files; ergonomics regression for `<SvgIcon id="…">` API                                                               |
| Per-icon `<img src="sprite.svg#id">`                                           | No JS code change                                                                                                                                                           | `<img>` fragment loading was the original Safari sprite bug (Smashing Mag, 2016); fragments still work poorly in some WebKit configurations                       |
| Convert problematic icons to PNG (extend `pngIcons` map)                       | Targeted; no architecture change                                                                                                                                            | Loses crispness; bundle size grows; doesn't scale (`opencascadejs` plus 6 brand icons = 7+ raster assets); already-rejected pattern for non-`manifold` cases      |
| Convert problematic icons to flatter SVG (no filter/mask)                      | Keeps sprite + `<use>` model                                                                                                                                                | Loses the official brand mark for `opencascadejs`; doesn't help `meta`/`autodesk`/etc. which legitimately need gradients                                          |

The inline-once approach is well-precedented (`octicons` / GitHub uses it for the same reason) and adds one app-shell-level component without touching every icon source.

### R1 — SMIL replacement for `LogoLoader`

```typescript
import { cn } from '#utils/ui.utils.js';

const TOP_REST = 'M256 66.6 L384.97 33.3 L481.7 58.28 L384.97 83.25 L256 116.55 L127.03 83.25 L30.3 58.28 L127.03 33.3 Z';
const TOP_FLEX = 'M256 0 L384.97 33.3 L481.7 58.28 L384.97 83.25 L256 49.95 L127.03 83.25 L30.3 58.28 L127.03 33.3 Z';
const LEFT_REST = 'M14.17 87.42 L62.54 99.9 L239.88 145.69 L239.88 462.05 L239.88 512 L191.51 499.51 L191.51 183.16 L14.17 137.37 Z';
const LEFT_FLEX = 'M14.17 87.42 L62.54 99.9 L62.54 416.26 L239.88 462.05 L239.88 512 L191.51 499.51 L14.17 453.73 L14.17 137.37 Z';
const RIGHT_REST = 'M497.83 87.41 L449.46 99.9 L272.12 145.69 L272.12 462.05 L272.12 512 L320.49 499.51 L320.49 183.16 L497.83 137.37 Z';
const RIGHT_FLEX = 'M497.83 87.41 L449.46 99.9 L449.46 416.26 L272.12 462.05 L272.12 512 L320.49 499.51 L497.83 453.73 L497.83 137.37 Z';

const EASE_IN_OUT = '0.85 0 0.15 1';

export function LogoLoader({ className }: { readonly className?: string }): React.JSX.Element {
  return (
    <div className={cn('size-4', className)} style={{ zIndex: 10 }}>
      <svg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 512 512' className='size-full'>
        <path fill='currentColor' d={TOP_REST}>
          <animate
            attributeName='d'
            dur='1s'
            repeatCount='indefinite'
            keyTimes='0;0.12;0.27;0.53;0.68;1'
            keySplines={`${EASE_IN_OUT};${EASE_IN_OUT};${EASE_IN_OUT};${EASE_IN_OUT};${EASE_IN_OUT}`}
            calcMode='spline'
            values={`${TOP_REST};${TOP_REST};${TOP_FLEX};${TOP_FLEX};${TOP_REST};${TOP_REST}`}
          />
        </path>
        <path fill='currentColor' d={LEFT_REST}>
          <animate
            attributeName='d'
            dur='1s'
            repeatCount='indefinite'
            keyTimes='0;0.05;0.20;0.46;0.61;1'
            keySplines={`${EASE_IN_OUT};${EASE_IN_OUT};${EASE_IN_OUT};${EASE_IN_OUT};${EASE_IN_OUT}`}
            calcMode='spline'
            values={`${LEFT_REST};${LEFT_REST};${LEFT_FLEX};${LEFT_FLEX};${LEFT_REST};${LEFT_REST}`}
          />
        </path>
        <path fill='currentColor' d={RIGHT_REST}>
          <animate
            attributeName='d'
            dur='1s'
            repeatCount='indefinite'
            keyTimes='0;0.19;0.34;0.60;0.75;1'
            keySplines={`${EASE_IN_OUT};${EASE_IN_OUT};${EASE_IN_OUT};${EASE_IN_OUT};${EASE_IN_OUT}`}
            calcMode='spline'
            values={`${RIGHT_REST};${RIGHT_REST};${RIGHT_FLEX};${RIGHT_FLEX};${RIGHT_REST};${RIGHT_REST}`}
          />
        </path>
      </svg>
    </div>
  );
}
```

`keyTimes` and `keySplines` mirror the original CSS keyframe percentages and the `cubic-bezier(0.85, 0, 0.15, 1)` easing.

### R2 — Inline-sprite delivery

Add a sprite mount component at the app shell:

```typescript
import sprite from '#components/icons/generated/sprite.svg?raw';

export function SvgSpriteMount(): React.JSX.Element {
  return (
    <div
      aria-hidden
      style={{ position: 'absolute', width: 0, height: 0, overflow: 'hidden' }}
      dangerouslySetInnerHTML={{ __html: sprite }}
    />
  );
}
```

Mount it once near the root of `apps/ui/app/root.tsx` (above `<Outlet />`). Then change `<SvgIcon>` to:

```typescript
<svg {...properties} className={className} viewBox='0 0 56 56'>
  <use href={`#${resolvedIconId}`} />
</svg>
```

Drop the `import spriteSvg from …` line. Every reference is now same-document, so Safari (and every other engine) materialises the full `<defs>` of each `<symbol>` and renders gradients/masks/filters correctly.

The SVGO pass already wrapped each icon in a `<symbol>` with its own `viewBox`, so the existing `viewBox='0 0 56 56'` on the wrapper `<svg>` continues to scale every symbol uniformly.
